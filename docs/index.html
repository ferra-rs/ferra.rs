Ferra.rs — User Guide

Getting started with Ferra

Ferra is a Rust framework for building APIs from model definitions. This page walks a contributor from a fresh clone of the repository to a green cargo check --workspace and a working test suite.

If you only have five minutes, read Building Ferra and Daily loop. Everything else is supporting detail.

Workspace layout

Ferra is a Cargo virtual workspace rooted at the repository root. The tree looks like this:

ferra/
├── Cargo.toml              # virtual workspace manifest
├── rust-toolchain.toml     # pinned Rust toolchain
├── .cargo/
│   └── config.toml         # short command aliases
├── crates/
│   ├── ferra-core/                    # the zero-I/O contract layer
│   ├── ferra-forge/                   # #[derive(FerraModel)] proc-macro (0.2.0)
│   ├── ferra-forge-trybuild-no-deps/  # CI-only trybuild host for the FR-021 missing-dep fixture (0.5.0)
│   ├── ferra-db/                      # repository layer over Sea-ORM (0.3.0)
│   ├── ferra-http/                    # Axum CRUD handler layer (0.4.0)
│   ├── ferra-openapi/                 # OpenAPI 3.1 + Scalar UI (0.5.0)
│   └── ferra/                         # facade re-export crate (0.4.0)
├── examples/
│   └── hello-ferra/        # canonical five-line CRUD reference (0.5.0)
├── docs/
│   ├── decision-record/    # architecture decision records (ADRs)
│   ├── roadmap/            # phased roadmap (0.1.0 → 1.0.0)
│   └── user-guide/         # this guide
└── pocs/                   # standalone proof-of-concept workspace (excluded)

0.4.0 Refining adds three workspace-level dep pins that are not on the constitution’s tracked-crate list: serde_json, tower-http (with cors / trace / limit features), and tower_governor. They exist to wire the ferra-http crate’s RFC 7807 body emission, the constitutional Tower layer stack (CORS / body limit / TraceLayer), and the per-route rate-limit layer on mutation endpoints respectively. Each carries an inline pin-rationale comment in the root Cargo.toml.

The pocs/ directory is its own independent Cargo workspace and is excluded from the main workspace. cargo check --workspace from the repo root builds only the framework crates, never the POCs.

Later phases add sibling crates under crates/ — ferra-http (0.4.0), ferra-openapi (0.5.0), ferra-auth (0.8.5), and ferra-anvil (0.13.0 CLI). 0.1.0 shipped ferra-core; 0.2.0 added ferra-forge; 0.3.0 adds ferra-db and the ferra facade crate.

The consumer Cargo.toml: one Ferra dep, one prelude opener

A consumer crate built on Ferra declares a single direct dependency among the ferra-* family AND no direct Sea-ORM dependency. The canonical 0.5.0 Rolling shape is:

[dependencies]
ferra = "0.5"
tokio = { version = "1", features = ["macros", "rt-multi-thread"] }
axum  = "0.8"
serde = { version = "1", features = ["derive"] }

That is the entire framework dep surface. ferra-core, ferra-db, ferra-forge, ferra-http, ferra-openapi, and sea-orm are all reached through the ferra facade — they MUST NOT appear as direct deps. The framework’s CI dep-shape gate enforces this on the two reference examples (examples/hello-ferra, docs/examples/movie-stars): a future regression that re-introduces any of those crates as a direct dep is a CI failure.

The one-line opener

Open every Ferra source file with a single glob over the prelude module:

#![allow(unused)]
fn main() {
use ferra::prelude::*;
}

That brings the dozen items a typical CRUD API needs into scope at once: the #[derive(FerraModel)] derive, the typed time newtypes (Date, DateTime), the typed error type (FerraError), the runtime state and router primitives (FerraState, ferra_router, Foundry), the typed-extraction wrappers (FerraJson, FerraPath), the response envelopes and pagination params (ItemResponse, CollectionResponse, PaginationParams), and Sea-ORM’s entity-derive prelude (DeriveEntityModel, EntityTrait, ColumnTrait, …) reached via cohabitation. serde::{Deserialize, Serialize} are also included so model structs declare their derive-attribute set with no extra imports.

Reaching Sea-ORM through the facade

The surrounding ORM is a cohabitation re-export at the facade root (ADR-0023). When you need the sea_orm::Database::connect(...) constructor, the #[sea_orm(...)] attribute namespace, or any other Sea-ORM type, write ferra::sea_orm::...:

#![allow(unused)]
fn main() {
let conn = ferra::sea_orm::Database::connect(&database_url).await?;
}

The use ferra::prelude::*; glob also brings the sea_orm module itself into scope, so the #[derive(DeriveEntityModel, FerraModel)] sibling-derive pair on a model struct works without a separate use sea_orm::entity::prelude::*; line. See ferra-db.md for the repository layer and ferra-forge.md for the model-derive grammar.

Crates.io publishing note

The ferra name on crates.io belongs to an unrelated project, so the framework’s facade is published as ferra-rs. Inside this workspace the directory is crates/ferra/ and the import path is ferra::*; when consuming the published crate, use the standard Cargo package alias to keep your import paths stable:

[dependencies]
ferra = { package = "ferra-rs", version = "0.5" }

The alias is invisible once set — all docs, examples, and Rust source in this guide write use ferra::*;.

Toolchain

The workspace pins a specific Rust release in rust-toolchain.toml:

[toolchain]
channel    = "1.94.1"
components = ["rustfmt", "clippy"]
profile    = "minimal"

The first time you run any cargo command inside the workspace, rustup reads this file and installs the pinned toolchain automatically. No manual rustup install is required.

Confirm the active toolchain with:

rustup show

The active toolchain line must report the version above. If it reports something else, an environment override is winning (check rustup override list).

The workspace also declares rust-version = "1.88" — this is the MSRV (minimum supported Rust version) that Ferra commits to for downstream consumers. The development toolchain pin and the MSRV serve different purposes: the pin sets the development + CI toolchain; the MSRV is the public contract for anyone depending on ferra-* crates.

Building Ferra

From a fresh clone:

git clone <repo-url> ferra
cd ferra
cargo check --workspace

On first run this fetches the pinned toolchain, downloads dependencies, and verifies every crate compiles. Subsequent runs are incremental.

Run the full test suite with:

cargo test -p ferra-core

The five-minute contract (success criterion SC-001 of the Excavation spec): a fresh clone reaches cargo test -p ferra-core green in under five minutes on a machine with internet access and a warm toolchain cache. If your first run is slower, the toolchain download is the culprit; the second run is the representative baseline.

Daily loop

The workspace declares three Cargo aliases in .cargo/config.toml:

AliasExpands toWhen you use it
cargo ccheck --workspaceAfter any edit — the fastest feedback loop.
cargo tnextest runRun tests through cargo-nextest.
cargo xexpandInspect expanded proc-macro output.

cargo t requires cargo-nextest to be installed (cargo install cargo-nextest --locked). If it is not available, fall back to cargo test. cargo x requires cargo-expand (cargo install cargo-expand --locked); it is rarely load-bearing in 0.1.0 but becomes the ground-truth lens for #[model] output from 0.2.0 onward.

Before committing:

cargo fmt
cargo clippy --workspace -- -D warnings
cargo test --workspace

All three must succeed. unsafe_code is forbid-ed at workspace level; a clippy warning is CI-blocking.

Pinned stack

The workspace declares a single shared version for every dependency every framework crate may use. Individual crates opt in by writing { workspace = true } in their own [dependencies]:

CrateVersionFirst consumed in
tokio1.51 (macros, rt-multi-thread)0.3.0
serde1 (derive)0.1.0
sea-orm=2.0.0-rc.38 (runtime-tokio-rustls, sqlx-postgres, macros)0.3.0
axum0.80.3.0
utoipa50.5.0
utoipa-scalar0.3 (axum)0.5.0
thiserror20.3.0
uuid1 (v4, serde)0.1.0
jiff0.2 (serde)0.5.0
tracing0.10.3.0
tracing-subscriber0.30.3.0
inventory0.30.10.5 (dormant)

The sea-orm exact-pin (=2.0.0-rc.38) is intentional: while Sea-ORM 2.0 is at release-candidate status, any RC bump is a deliberate act and never a resolver side effect. Full rationale lives in the ADRs — those cross-references are non-load-bearing sidenotes; this table alone is sufficient for building Ferra.

ferra-core itself depends on only two of these: serde and uuid. That cap is part of the 0.1.0 contract.

ferra-forge (0.2.0) is proc-macro-only and pulls in four compile-time deps that are not in the table above — syn, quote, proc-macro2, heck — plus trybuild as a dev-dep for its proc-macro UI test suite. None of these are part of the runtime contract; consumers never link them.

Pinned stack: sidenote on raw SQL

You will notice sqlx and sea-query are not in the table above. They are pulled in transitively through sea-orm but are not part of Ferra’s public contract surface. A reader who expects to use ferra::sqlx::* will not find that path — it does not exist and is not planned. If you need raw SQL in an application built on Ferra, reach through Sea-ORM’s own escape hatch inside your application code; the framework does not re-export it.

sqlx DOES appear as a dev-dependency of ferra-db from 0.3.0 onward — it powers the #[sqlx::test] integration harness (plus the SqlxPostgresConnector::from_sqlx_postgres_pool wiring that hands Sea-ORM the isolated per-test database). It is test-only, never a runtime dep, and therefore never ships to downstream consumers.
